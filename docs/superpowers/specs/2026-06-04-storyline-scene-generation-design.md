# SceneSelf 故事线场景生成 — 设计文档

- 日期:2026-06-04
- 状态:已批准,待实现
- 范围:SceneSelf 核心出图链路(场景规划 + 问答 + 出图),不含支付/认证

## 1. 背景与问题

用户实测反馈两个核心问题:

1. **缺乏故事性**:一组 6 张照片常有 5 张几乎是同一场景,只是人物动作/表情/角度不同。例如"豪华游轮看鲸鱼"生成的 6 张都是甲板。
   - 根因 A:上一版的"叙事弧"(approach→boarding→cockpit→…)本质是**同一主场景内的动作时间线**,不是不同场景。
   - 根因 B:每帧 image_prompt 的主视觉是同一个地点(如"游轮甲板"),Gemini 出图必然同质化。

2. **问答问错了维度**:当前问的是氛围/光线/风格(vintage/film/aesthetic),既不帮助构建故事,又偏离产品核心。

## 2. 核心理念

**一组照片 = 一段体验的故事线 = 6 个不同时刻/场景,合起来汇总成用户输入的完整体验。**

从"同场景多角度/多动作"彻底转向"多场景叙事"。产品核心竞争力是"**记录一段真实体验**",不是滤镜风格(竞品在卷的)。

示例:"豪华游轮看鲸鱼" + [惊喜高光] + [自然奇观] →
> 🛳️登船震撼 → 🥂甲板香槟 → 🦞餐厅海鲜 → 🐋看鲸鱼(高潮) → 🌅船头自拍 → 🎉夜晚派对(换装)

每个是不同的活动/地点/视角,但同一个人、同一段连续旅程。

### 2.1 故事线的组织逻辑随内容类型变化(关键)

"故事线"不是单一的"时间线旅程"。6 个场景始终要"明显不同",但**"6 个场景靠什么串联"取决于内容类型**。这复活并扩展了项目原有的 `coherence_type`。

**故事线类型库(8 类主原型 + AI 归类 + 通用兜底):**

| # | 类型 | 典型用户输入 | 组织逻辑(6 场景靠什么串) | 连贯锁定 |
|---|---|---|---|---|
| 1 | 旅程体验 | 游轮/旅行/探店/约会/citywalk | 时间线:按时间推进的不同时刻 | 同一段旅程 + 主造型 |
| 2 | 拥有炫耀 | 买飞机/豪车/游艇/名表/豪宅/宠物 | 物体环绕:围绕核心物体的不同场景,凸显"我的" | 物体外观 + 主造型 |
| 3 | 幻想角色 | 穿越古代/未来、变超人/动漫/电影主角 | 名场面集锦:同一身份在不同情境的高光 | 角色造型 + 世界观 |
| 4 | 高光事件 | 毕业/婚礼/生日/升职/获奖/开业 | 事件弧:准备 → 仪式高潮 → 欢庆 | 同一场事件 + 主造型 |
| 5 | 职业身份 | CEO/医生/飞行员/厨师/艺术家 | 身份切面:工作/专注/休息/高光不同侧面 | 职业装 + 场所 |
| 6 | 生活美学 | 慢生活/健身/居家/晨间/咖啡日常 | 主题变奏:同一美学的不同时刻 | 美学基调 + 主造型 |
| 7 | 节日季节 | 圣诞/万圣节/樱花季/雪景/新年 | 氛围铺陈:节日的不同场景与活动 | 节日元素 + 主造型 |
| 8 | 蜕变成长 | 减肥蜕变/变装/造型转变 | 对比弧:从一个状态到另一个状态 | 同一人 + 转变主题 |

- 边缘/混合内容(如"古风旅拍" = 旅程 + 幻想):AI 归到**最匹配主类 + 融合元素**,不强行互斥;归不进的走**通用多场景兜底**。
- 不无限扩张类型(类型越多 AI 分类越易错)。

**每类驱动专属问答**:分类结果决定 Q1 调性预选倾向 + Q2 侧重选项。例:
- Q2 侧重:旅程→风景人文/美食/购物/悠闲;炫耀→极致豪华/生活方式/社交名利/质感细节;幻想→世界观沉浸/角色魅力/史诗大场面/唯美意境。
- 调性预选:炫耀类默认高亮「凡尔赛」,幻想类高亮「电影戏剧/燃系热血」。

### 2.2 陪伴修饰维度(关系类的轻量实现)

"关系/陪伴"**不是独立类型,而是正交修饰维度**,可叠加在任意类型上("和闺蜜游轮游" = 旅程 + 陪伴)。

- **实现**:第二人只出现**背影/侧影/牵手入镜/虚化**、**不露正脸**,作为"场景氛围元素",不做人物一致性保证。
- 传达"有人陪伴"的氛围即可,规避多人一致性这个行业难题。
- **不做**:真双人合照(上传第二张脸、锁两人跨 6 张),单独立项(见第 8 节)。

### 2.3 类型感知的造型 / 道具 / 视角(v2 补充,实测后)

实测发现:故事线(场景)对了,但人物的**造型、道具、拍摄方式没跟着故事线类型走** —— 出现"古代背景里的现代人在自拍"(穿现代红毛衣牛仔裤、背现代包、举手机)。根因:continuity(造型)硬编码现代休闲装 + selfie 视角不真实。

**8 类风险地图:**

| 类型 | 造型该是 | 现代道具(手机/包) | 自拍 | era |
|---|---|---|---|---|
| 旅程体验 | 现代装 | 可 | 可 | modern |
| 拥有炫耀 | 现代奢华装 | 可 | 可 | modern |
| 幻想角色 | 角色服(古装/战衣/未来) | 🚫 禁 | 🚫 禁 | historical/fantasy/future |
| 高光事件 | 正式装(婚纱/学士服/礼服) | 正式场合禁包 | 可 | modern |
| 职业身份 | 职业制服(白大褂/厨师服/机长服) | 职业道具 | 工作场景少 | modern |
| 生活美学 | 现代日常/运动装 | 可 | 可 | modern |
| 节日季节 | 季节/节日装(冬装/cosplay) | 看场景 | 可 | modern |
| 蜕变成长 | 造型要"变化"(前后不同) | 看 | 可 | modern |

**四个约束维度(规避方案):**

1. **造型由 LLM 按类型生成**(核心):故事线生成时,LLM 据 [类型 + 内容] 一起产出 set-level continuity(outfit/hair/accessory)。古代→盔甲战袍、医生→白大褂、婚礼→婚纱、雪景→冬装。取代硬编码 activity styling(后者降级为无 LLM 时的 fallback)。
2. **era 标签 + 约束**:每个 `StorylineTypeDef` 标 `era`(modern/historical/fantasy/future)+ `allowSelfie`/`allowModernProps`。historical/fantasy → 强制**禁 selfie + 禁现代道具**(手机/现代包/运动鞋);职业/正式 → 禁现代包。约束注入故事线生成 prompt(LLM 不给古代安排自拍)+ 每帧 prompt negative。
3. **selfie 真前置视角**(全类):画面**看不到拿手机的手**、无伸向镜头的手臂;减少 selfie 占比,多用 friend_candid(朋友帮拍、不拿手机)。
4. **transformation 特例**:允许 1-2 帧造型递进(起点 → 蜕变后),不锁死主造型。

**质检配套**:收紧 deformity 判断,多手/多肢(如自拍三只手)必剔除。

## 3. 已确认的设计决策

### 3.1 问答:2 题(调性 + 侧重)
- **Q1 故事调性**(灵魂):探测用户想要的"感觉/调性"。
- **Q2 体验侧重**:据用户输入动态生成 4 个选项。
- 砍掉所有 vintage/film/光线/aesthetic 风格问题。
- **拍摄视角、节奏跨度等交给 AI 自动判断**(不问用户,越快越好)。

### 3.2 调性机制:AI 预选 + 8 精选 + Other
- **AI 预选高亮**:用户输入后,AI 据内容预判并高亮 1-2 个最匹配调性(降低决策负担,呼应"AI 感知意图")。
- **8 个精选调性库**(覆盖高频):叙事纪实 / 惊喜高光 / 松弛治愈 / 电影戏剧 / 凡尔赛炫耀 / 搞笑沙雕 / 浪漫氛围 / 燃系热血(具体名单实现时可微调)。
- **Other 自由描述**:长尾个性化,用户打一句话(如"复古港风"),AI 理解文本生成对应调性(无限丰富,不占屏)。

### 3.3 UI:移动端优先,方案 A 全平铺 2 列卡片
- 调性页:8 个 emoji 卡片 2 列全平铺,AI 推荐的带「AI」角标高亮 + Other 兜底。
- 问答流程:**一屏一问 + 顶部进度条**(共 2 题),末题按钮"生成我的故事"。
- 不横滑(会藏选项)、不全屏铺。

### 3.4 一致性:主造型 + 关键场景换装
- 锁人物的脸(reference chaining)。
- **主造型**贯穿大部分场景(一眼认出是同一段旅程)。
- 允许 **1-2 个关键场景合理换装**(如晚宴礼服、泳池泳装)。

### 3.5 拍摄视角:AI 自动按场景分配
- 每个场景标注 `selfie`(自己举手机自拍感)或 `friend_candid`(朋友抓拍、当事人恰好在画面)。
- 看鲸鱼→朋友抓拍;船头→自拍。增强"真实记录"感。

## 4. 技术架构

### 4.1 两段式故事线生成(核心改动)

取代现有 `scene-plan.ts` 的硬编码 `narrativeBeats` 骨架(`VEHICLE_ARC` / `OBJECT_ARC` / `GENERIC_ARC`)。

**第零步 — 分类(复活并扩展 `coherence_type`)**:AI 据用户输入归到 8 类故事线类型之一(可融合元素,归不进走通用兜底),决定后续的组织逻辑、连贯锁定与专属问答选项。

**第一步 — 故事线生成(LLM)**:
输入 `[体验描述 + 调性 + 侧重 + 故事线类型]` → 输出一条故事线 = 6 个**互不相同的子场景**(按该类型的组织逻辑组织,见 2.1)。
硬约束:6 个场景的**地点 / 活动必须明显不同**(不是同一地点的不同角度);其中 1 个为"高潮/惊喜"节拍呼应调性。

`StoryBeat` 数据结构:
```ts
interface StoryBeat {
  index: number;
  scene_title: string;        // "鲸鱼浮出喷水的瞬间"
  setting: string;            // 具体地点/环境
  activity: string;           // 在干什么
  shot_perspective: 'selfie' | 'friend_candid';
  shot_size: 'wide' | 'medium';
  wardrobe: 'main' | string;  // 'main' 或 "change:晚礼服"
  expression_beat: string;    // 本帧微表情
  is_highlight?: boolean;     // 高潮帧
}
```

**第二步 — 展开每帧 image_prompt**:
把每个 `StoryBeat` 展开成详细出图 prompt:场景描述用本 beat 的 `setting`+`activity`(不再用整体 safePrompt),注入 continuity 锁定、视角(自拍/朋友拍措辞)、换装描述,统一手机随手拍风格。

### 4.2 出图与一致性
- **reference chaining 保留**:先出第 1 帧(主造型基准)→ 作为额外参考喂后续帧。
- **换装帧**:reference chaining 对换装帧造型一致性帮助有限,靠 prompt 明确描述新造型 + 锁脸维系连贯。
- 统一"普通手机随手拍"质感:deep focus、无 bokeh、真实皮肤纹理、自拍/朋友抓拍的不完美构图。

### 4.3 调性与侧重的实现
- **调性库**:8 个常量,每个含 `id / label / emoji / promptFragment`(注入故事线生成的调性指令)。
- **AI 预选**:一次轻量 LLM 调用(可并入现有 classify 步骤),据输入推荐 1-2 个调性 id。
- **Q2 侧重**:据输入动态生成 4 个选项(LLM)。

## 6. 测试策略
- 单元:故事线生成产出 6 个 distinct `setting`/`activity`;高潮帧存在;调性预选映射;每帧 prompt 含本场景 `setting`(不再只是整体主题)。
- 回归:保留现有 reference chaining、质检、退款、积分测试。
- 端到端:实测多类场景(游轮/旅行/美食/载具/日常)验证故事性与一致性。

## 7. 风险与权衡
- **一致性 vs 场景多样**:场景全变 + 换装增加"看出同一人"难度。缓解:主造型锁定 + reference chaining + 质检 same_person 兜底。
- **Vercel 300s**:两段式多一次 LLM 调用(Gemini Lite ~2-3s,可接受)。故事线 + 6 帧出图 + reference chaining 第 1 帧串行,需监控总耗时;必要时降并发/关 reference chaining。
- **故事线质量依赖 LLM**:第一步生成 prompt 是关键,需精心设计 + 强约束"6 个场景必须明显不同 + 1 个高潮"。fallback 路径(无 key/解析失败)需要一个通用的"多场景"骨架兜底。

## 8. 不做的(YAGNI)
- **故事线预览/编辑页**:用户选"无感知直接生成",不向终端用户展示故事线。
- **风格/滤镜系统**:明确砍掉。
- **多于 2 题的问答**。
- **真双人合照**(上传第二张脸、锁两人跨 6 张):技术难、失败率高、耗时翻倍,**单独立项**;本次关系类只做"陪伴修饰"(第二人不露正脸,见 2.2)。

## 9. 迁移说明
- 现有 `narrativeBeats` / `VEHICLE_ARC` 等相关测试需相应改写或移除。
- 现有 `applyPlanPostProcessing` 的"载具叙事弧强制覆盖"逻辑被故事线机制取代。
- continuity 的 user color override、activity styling 等仍保留(它们锁的是"主造型",与故事线正交)。
