## 架构

积分系统使用**双写模式**进行余额跟踪：

- `user.credits` — 快速访问的当前余额（用于快速读取）
- `creditLedger` — 不可变的积分变动审计记录

两者始终在同一个数据库事务中更新，确保一致性。

## 核心 API

所有函数在 `lib/credits.ts` 中：

```typescript
// 查询用户余额
const credits = await getUserCredits(userId);

// 检查用户是否负担得起
const canAfford = await canUserAfford(userId, 300);

// 扣除积分（事务性）
await deductCredits(userId, 300, 'scene_set');

// 退还积分（如有照片未交付）
await refundCredits(userId, 50, 'scene_refund');
```

## 积分消耗

| 操作 | 消耗 | 配置位置 |
|------|------|---------|
| 场景写真一组（6 张） | 6 × 50 = 300 积分 | `lib/scene/pricing.ts`（`SHOTS_PER_SET`、`CREDITS_PER_PHOTO`） |

任务启动时预扣全组积分；任务结束时未交付的张数按张返还。

## 积分来源

| 来源 | 触发条件 | 数量 |
|------|---------|------|
| 注册奖励 | 新用户注册 | 300 积分 |
| 订阅 | Webhook / 定时任务 | 按计划配置 |
| 兑换码 | 用户在 `/redeem` 核销 | 按兑换码配置 |
| 管理员调整 | 通过管理后台手动操作 | 自定义 |

## 账本原因

每条 `creditLedger` 记录都有 `reason` 字段：

- `registration_bonus` — 注册赠送
- `scene_set` — 场景写真组扣除（300 积分）
- `scene_refund` — 未交付照片返还
- `subscription_cycle` — 订阅周期发放
- `redemption` — 兑换码积分
- `admin_adjustment` — 管理员手动调整

## 积分补偿

场景生成使用**先扣除、失败后退还**模式：

```typescript
// 1. 任务启动时预扣全组积分
await deductCredits(userId, 300, 'scene_set');

// 2. 执行生成任务
// ...

// 3. 任务结束时返还未交付的张数
await refundCredits(userId, refundForUndelivered(delivered), 'scene_refund');
```

确保照片未交付时用户不会损失积分。
